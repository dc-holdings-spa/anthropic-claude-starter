# Commit message style

Cuando se solicite generar un commit message:

## Formato

```
<type>(<scope>): <subject>

<body>

<footer>
```

## Reglas

1. **Subject ≤50 chars**, imperativo presente: "add X" no "added X"
2. **Type**: `feat` | `fix` | `refactor` | `docs` | `test` | `chore` | `perf` | `style`
3. **Scope**: módulo afectado (`api`, `rag`, `eval`, `mcp`, etc.)
4. **Body**: solo si el "why" no es obvio del diff. Máximo 3 líneas.

## Ejemplos

```
fix(caching): preserve cache_control on list system blocks

Previously the wrapper overwrote existing cache_control flags when
caller passed a pre-structured system list. Now respects caller-set
control and only adds breakpoint when missing.
```

```
feat(rag): hybrid retrieval con RRF

Vector (Voyage) + BM25 + Reciprocal Rank Fusion. Default k=60.
Rescata exact-match codes que vector solo missea.
```

## Reglas extra

- Cero emojis salvo que user pida
- Cero "Generated by Claude" suffix
- Cero referencias a "based on user request"
- Si hay breaking change → `feat!:` o `BREAKING CHANGE:` en footer
